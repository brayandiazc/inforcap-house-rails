# Convenciones de internacionalización (i18n)

> Cómo manejamos múltiples idiomas y locales en InforcapHouse.
> **Última actualización**: 2026-07-02

## Locales soportados

- **Por defecto**: `en` (locale por defecto de Rails; archivos en `config/locales/`).
- **Adicionales**: ninguno por ahora.

> Estado actual: el MVP es **prácticamente monolingüe**. El contenido de las vistas está en español (mayormente escrito directo en los ERB) y el framework de i18n usa `en` por defecto. Si se formaliza el multiidioma, añadir `config/locales/es.yml` y fijar `config.i18n.default_locale = :es`.

## Detección de locale

Orden de prioridad sugerido:

1. Preferencia explícita del usuario (perfil/cookie).
2. Cabecera `Accept-Language`.
3. Locale por defecto.

## Reglas

- Ningún texto visible al usuario se escribe hardcodeado: todo pasa por el sistema de traducción.
- Las claves de traducción se organizan por dominio/feature, no por pantalla.
- Mantener **paridad** entre locales: una clave existe en todos los idiomas soportados.
- Formatos de fecha, número y moneda se localizan según el locale activo.

## Estructura

```text
config/locales/
├── en.yml
├── devise.en.yml
└── ...            # es.yml al internacionalizar
```

## Ejemplos

```text
# clave de traducción
usuarios.bienvenida: "Hola, %{nombre}"
```

## Comandos útiles

```bash
# Recomendado al haber múltiples locales:
bundle exec i18n-tasks health   # Verifica claves faltantes/sin usar
```
